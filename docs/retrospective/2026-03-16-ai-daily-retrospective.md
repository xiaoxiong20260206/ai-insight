## 三、根因分析

### 问题1：内容严重缩水（核心问题）

**表象**：185行 → 应该是460+行，7条 → 应该是18条

**直接原因**：
- 在修复"时效性检查失败"时，删掉了大量超出时间窗口的新闻
- 没有用新的窗口内新闻替代，而是直接删除

**根因**：
1. **错误的修复策略**：选择"删减"而非"替换"
2. **缺乏参照基准**：没有与昨天日报对比内容量
3. **质量门盲区**：质量门只检查"格式正确"，不检查"内容充足"

**类比**：
> 病人发烧39°C，给他敷冰块降到35°C ——症状消失了，但病人快冻死了。
> 正确做法是找病因对症下药，不是强行降温。

---

### 问题2：JSON结构缺失

**表象**：缺少 `data_snapshot`、`watch_list`、`references`、`capability_update`

**直接原因**：
- 第一次生成JSON时使用了错误的格式（参照搜索结果而非昨天模板）
- 第二次重写时只关注了`tabs`结构，忽略了其他顶级字段

**根因**：
1. **模板意识薄弱**：没有"对照标杆"的习惯
2. **增量思维陷阱**：只关注修复错误，不关注保留正确

---

### 问题3：链接不可达（3次替换）

**时间线**：
1. UNICEF链接不可达 → 替换为A2 Collective
2. Reddit链接不可达 → 替换为Builder.io
3. 知乎链接不可达 → 替换为中新网

**根因**：
1. **搜索结果直接使用**：没有在添加前验证链接可达性
2. **高风险来源**：Reddit/知乎等来源稳定性差

**教训**：
> 链接来源优先级：官网 > 权威媒体 > 科技媒体 > 社区（Reddit/知乎）

---

### 问题4：JSON格式损坏

**表象**：`multi_replace_in_file` 替换后JSON语法错误（第262行）

**根因**：
- 批量替换时没有完整匹配上下文
- 替换后没有立即验证JSON语法

**教训**：
> 每次编辑JSON后，必须立即运行 `python3 -c "import json; json.load(...)"`

---

### 问题5：字段名错误

**表象**：`data_snapshot` 使用了 `label/value/change/source`，脚本期望 `metric/value/note`

**根因**：
- 参照了错误的数据结构（可能是其他项目的格式）
- 没有查看昨天的实际字段名

**教训**：
> 新增模块时，必须先读取昨天的同名模块确认字段契约

---

### 问题6：URL重复

**表象**：3个URL在不同tab中重复出现

**根因**：
- 同一个新闻源（NVIDIA官博/新浪）被用于多条不同角度的新闻
- 没有意识到质量门会检测跨tab URL重复

**教训**：
> 同一URL只能出现在一个tab，不同角度的报道需使用不同来源

---

### 问题7：Git提交遗漏

**表象**：KIM推送完成后，内外部仓库都没有git push

**根因**：
1. **流程意识断裂**：认为"KIM推送成功=任务完成"
2. **6处联动认知不全**：知道更新文件，不知道还要push

**教训**：
> 日报完成的标志是 **Git push成功 + GitHub Pages生效**，不是KIM发送成功

## 四、模式识别

### 模式A：过度修复综合症

```
问题 → 修复 → 引入新问题 → 再修复 → 引入更多问题
```

本次案例：
- 修复"时效性" → 内容过少
- 修复"内容过少" → JSON格式错误
- 修复"JSON格式" → 字段名错误
- 修复"字段名" → URL重复

**根因**：每次修复只关注当前错误，不评估修复的副作用

### 模式B：最后一公里松懈

```
Step 1-4 认真执行 → Step 5 KIM推送 "收尾" → 忘记Git提交
```

**根因**：心理上把"发送KIM"当作终点，实际终点是"部署生效"

### 模式C：模板漂移

```
正确模板（昨天） → 错误理解 → 偏离模板 → 质量门失败
```

**根因**：没有建立"每次生成前先读昨天"的强制习惯

```python
# 建议新增的质量门检查项
def check_content_volume(data, reference_date):
    """对比参考日期的内容量，偏差超过30%报警"""
    ref_data = load_json(f"daily-content-{reference_date}.json")
    current_news = count_news(data)
    ref_news = count_news(ref_data)
    if current_news < ref_news * 0.7:
        return False, f"内容量不足：{current_news}条 vs 参考{ref_news}条"
    return True, None
```

## 七、核心教训

> **1. 修复不是删除**
> 质量门报"时效性失败"，正确做法是"找窗口内新闻替换"，不是"删掉超窗新闻"。

> **2. 模板是锚点**
> 每次生成新内容前，先读昨天的成品。偏离模板=偏离正确。

> **3. 终点是部署生效**
> KIM发送≠完成。Git push + GitHub Pages生效才是真正的终点。

> **4. 修复要评估副作用**
> 每次修复后，对比修复前后的"其他指标"是否正常。
