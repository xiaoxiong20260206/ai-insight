> 基于《【林克的AI洞察】可能99%的AI产品都做错了》及《你的同事正在被一个个「蒸馏」成 AI》提炼
> 核心：有立场、有温度、有类比、有金句、有真实故事引子

### 标杆B：深度调研型（组织护城河）

### 0.1 开头：真实故事引子（必须）

**不是理论铺陈，不是行业背景，是一段让人"卧槽"的真实场景。**

标杆做法：
- 虚构但高度可信的故事场景（老张离职但Agent还在回消息）
- 或真实事件引用（Anthropic挖其他公司CTO）
- 2-3段就把读者拉进场景
- 最后一句制造认知冲突："人走了，活没停。这不是科幻，是2026年正在发生的事。"

**禁止**：
- ❌ 以"在当今XX快速发展的时代"开头
- ❌ 以"本文将会介绍XXX"开头
- ❌ 以背景介绍/行业定义开头

### 0.2 标题结构（深度调研专用）

**深度调研**文章的标题和开头格式（从标杆B·沈浪修改定稿版学习）：

```markdown
# 【林克的AI洞察】主标题
## 副标题（一句话说清核心观点）
🌐 Web 交互版：强烈推荐访问 👉 [链接](URL)

# 📖 真实故事 · 引子
[引子正文...]

# 00 全文概览：先说结论
![配图](CDN链接)
[概览正文...]
```

**关键规律（沈浪修改定稿版确认）**：
- 标题两行：`# 【林克的AI洞察】主标题` + `## 副标题`
- 第二行是Web交互版链接
- **没有封面大图**——引子章节紧跟标题/Web链接，不需要独立封面图
- 配图在章节标题后面（不是标题前面），紧跟`# XX`标题下一行
- 章节之间无分割线（`---`），完全靠`# XX`标题分隔

### 0.3 00概览章：核心范式对比表+剧透金句

标杆做法：
- 一句话定义事情本身（中性描述）
- 一个核心问题让读者选择
- 核心范式对比表（两种选择/策略/态度）
- 📌剧透金句：告诉读者结论，但不展开

示例结构：
```
一件正在发生的事：[中性描述]
它带来了一个问题：[核心选择题]
[核心范式对比表]
📌 剧透：[简短结论，暗示有中间路线]
```

### 0.4 配图位置：紧跟章节标题后面（不是标题前面）

**核心格式规律（沈浪修改定稿版确认）**：

- 每个一级章节（01-N、09总结等）标题后面，紧跟一张信息大图
- **注意**：📖引子章节不需要配图
- 配图紧跟在`# XX`标题下一行，标题和图片之间没有空行
- 图片用 `![描述](CDN链接)` 格式
- 图片描述简洁：`![全文概览：两种护城河理解的对比]`

**正确格式**：
```markdown
# 01 三件事正在同质化——AI行业看得见的部分全在趋同
![三件事正在同质化：AI行业外显部分的收敛趋势](CDN链接)
Gupta观察到三个收敛趋势：
...
```

**错误格式（我原来写的，不要再用）**：
```markdown
# 01 三件事正在同质化——AI行业看得见的部分全在趋同


![配图](CDN链接)


Gupta观察到三个收敛趋势：
...
```

**章节编号规范**：
- 📖 引子：`# 📖 真实故事 · 引子`（emoji前缀）
- 00概览：`# 00 全文概览：先说结论`
- 正文章节：`# 01 三件事正在同质化——副标题`（两位数字+空格+主标题+——副标题）
- 总结：`# 09 总结：主标题`
- 彩蛋：`# 彩蛋：写这篇调研的林克`（无编号）

### 0.5 表格格式：空列头风格（但第一列有实际内容时不要用空列头）

**大多数表格用 `| | | |` 空列头风格**（从标杆B学习）：

```markdown
|  |  |  |
| --- | --- | --- |
| 内容A | 内容B | 内容C |
```

**但当第一列是分类维度时，用有意义的列头**：

```markdown
| 维度 | 可模仿的部分 | 不可模仿的部分 |
| --- | --- | --- |
| 制度 | DRI→业界传播 | 信念体系 |
| 角色 | FDE→全行业复制 | 世界观 |
```

**原理**：KIM Doc中空列头渲染更清爽，但当第一列本身就是维度分类时，空列头反而让人不知道每行代表什么。

### 0.6 📌标注规范

- 每章最多1-2次📌
- 📌只用于**反常识判断**或**关键结论**
- 格式：`📌 核心结论内容`（不加粗，让📌本身做标识）
- 不用于常识性陈述

### 0.7 正文章节：先概念→再拆解→再对比→📌点睛

标杆做法：
- 先解释一个关键概念（隐性知识）
- 再拆解结构（七层知识模型）
- 再对比两种态度（防御派vs拥抱派）
- 📌标注只在"反常识判断"或"关键结论"处使用，每章最多1-2次

### 0.8 行动清单章节：P0/P1/P2分级+具体操作（日报型专用）

标杆A做法：
- 用🔴🟡🟢标注优先级
- 每条行动有"为什么值得做"的解释
- 最简版操作模板（四个问题，10分钟）

**深度调研型一般不含行动清单**，聚焦于解读和洞察。

### 0.9 总结章节：从→到转变表+金句

标杆做法：
- 从→到转变表（旧规则→新规则）
- 最后一句金句：简洁（15字内）、有对比、可独立引用

### 0.10 彩蛋：自我印证

标杆做法：
- "这篇文章是林克写的。林克本身就是[论点]的活标本"
- 具体描述：怎么做的→每个工作循环都在更新知识资产
- 📌结尾金句：把论点映射回现实行动

### 0.11 了解更多模块（固定结尾）

每个KIM Doc结尾必须包含三条链接：


### 0.13 段落排版：紧凑不松散（沈浪修改定稿版确认）

**核心规律：KIM Doc排版要紧凑，不要松散**

- 段落之间0空行（紧凑排版），不要用空行分隔段落
- 关键词不加粗（`**加粗**`极少使用），让📌本身做标识
- 段落内用纯文本而非加粗强调，信息密度靠内容本身
- 章节之间无分割线（`---`），完全靠`# XX`标题分隔

**错误做法（我原来写的）**：
```markdown
他们来了，因为一句话：**"如果你想做真正安全的AI..."**

**找不到。这就是组织形态护城河的物理载体。**


![配图](CDN链接)


# 00 全文概览

核心论点：**当技术趋同...**
```

**正确做法（沈浪修改定稿版）**：
```markdown
他们来了，因为一句话："如果你想做真正安全的AI..."

找不到。这就是组织形态护城河的物理载体。

# 00 全文概览

![配图](CDN链接)
核心论点：当技术趋同...
```

**差异总结**：去加粗、去多余空行、配图紧跟标题后面

| 维度 | 日报型 | 深度调研型 |
|------|--------|-----------|
| 章节数 | 6-8章 | 10-11章（含引子+00+正文+总结+彩蛋） |
| 引子 | 真实故事 | 真实故事（更简短） |
| 行动清单 | 🔴🟡🟢分级 | 无（聚焦解读） |
| 章节编号 | 01-06+彩蛋 | 📖引子+00概览+01-N正文+09总结+彩蛋 |
| 配图 | 每章1张 | 每个一级章节1张+封面1张 |
| 反证/修正 | 无 | 有（VRIO等框架做反证） |
| 表格密度 | 中 | 高（几乎每章都有） |

## 一、整体风格定位

| 维度 | 规范 |
|------|------|
| **人称** | 第一人称（我、我们），偶尔「我们看到」「我认为」 |
| **立场** | 鲜明，有判断，不做「一方面另一方面」的两面派 |
| **语气** | 自信但不傲慢，严谨但不学究，幽默但不油腻 |
| **类比** | 大量使用生活化类比，让抽象概念具体可感 |
| **节奏** | 短段落为主（2-4句），重要观点单独一段加强 |
| **彩蛋** | 比正文轻松，允许口语化，要有「有意思」的发现感 |

## 二、文章开头写法

**好的开头 = 制造认知冲突**

```markdown
# 示例（参考文档开头）

2022年底，ChatGPT上线。没有发布会，没有广告，一条推文：

**"我们推出了ChatGPT，来试试吧。"**

5天后，用户破百万。60天后，破亿。

同一时期，国内某AI产品花了数千万买量、签KOL。3个月后，日活不到注册用户的3%。
```

**模式**：具体案例A（反常识的成功）→ 具体案例B（符合常识的失败）→ 一个问题

**禁止**：
- ❌ 以「在当今AI快速发展的时代」开头
- ❌ 以「本文将会介绍XXX」开头
- ❌ 以背景介绍/行业定义开头

## 三、比喻使用规范

> 参考文档核心比喻：「买电动工具 vs 招员工」

**好比喻的标准**：
1. 来自日常生活（工具、员工、学生、餐厅…）
2. 映射关系清晰（A的属性 = B的属性）
3. 能衍生出多个子映射（一个比喻能解释多个现象）

**常用比喻类别**：

| 场景 | 可用比喻 |
|------|---------|
| AI产品 vs SaaS | 员工 vs 工具，摩托车 vs 自行车 |
| 质量阈值 | 菜品难吃 vs 刚刚好吃 vs 惊艳好吃 |
| 先推广再打磨 | 厨师还没学会做菜就开门迎客 |
| 场景聚焦 | 先把一个菜做到极致，再开第二道 |
| 信任积累 | 银行存款，不能立即提现 |
| 技术栈 | 编码是底层OS，其他能力是App |

**类比展开模板**：
```
[比喻名称] = [简要描述]

[具体展开：比喻是怎么运作的]

这个比喻，解释了[主题]所有的[特征]：

[对比表格 — 对比A和B在各维度的映射关系]
```

## 四、表格写法规范

### 4.1 表头要精简有力

| 差 | 好 |
|----|-----|
| 「用户的核心需求类型」 | 「用户要什么」 |
| 「产品失败的典型表现形式」 | 「典型错误」 |
| 「阶段性工作的关键核心原则」 | 「关键原则」 |

### 4.2 数据单元格要具体

| 差 | 好 |
|----|-----|
| 「有价值」 | 「直接省脑力，不用再想了」 |
| 「不好」 | 「会主动劝退，告诉周围人'不行'」 |
| 「重要」 | 「Day 1就要付费，顶级员工不需要试用期」 |

### 4.3 结论标识符（📌）用法

```markdown
# 正确用法 — 每章最重要的洞察，用📌标出

📌 **核心结论**：AI产品有一条"质量阈值线"，低于阈值任何推广都是在加速负口碑传播。

# 对比规则：
- 每章最多用1-2次📌
- 📌后跟 **加粗的洞察标题**：洞察内容
- 📌标注的内容必须是「反常识的」或「关键判断的」，不是常识性陈述
```

### 4.4 产品评分表标准

```markdown
| 产品 | Q 价值密度 | E 成本收支 | T 信任 | D 聚焦 | 结论 |
|------|-----------|-----------|--------|--------|------|
| Cursor | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★★ | ✅ 教科书 |
| Claude Code | ★★★★★ | ★★★★ | ★★★★ | ★★★★★ | ✅ 教科书 |
| 某产品 | ★★★ | ★★ | ★★ | ★★ | ⚠️ 通用陷阱 |
```

**评分维度说明**：
- Q（质量/价值密度）：用完是否有「省了自己脑力」的感知
- E（经济性）：是否 Day 1 有付费，单位成本 vs LTV 是否合理
- T（信任）：是否有个性化记忆、是否有口碑传播
- D（聚焦度）：核心场景是否足够深，不是广铺功能

**结论标识**：
- ✅ 教科书
- ✅ 方向正确
- ⚠️ 验证了需求（但还有问题）
- ⚠️ 通用陷阱
- ❌ 需要反思

## 五、结论章节写法

### 5.1 三个转变（标准结构）

```markdown
| 从 | 到 |
|----|-----|
| "我们的AI有XX功能" | "我们的AI在XX任务上比用户自己做得好" |
| "先拉用户再打磨" | "先穿越阈值，再规模分发" |
| "用户量是北极星" | "用户复用率 + 付费意愿才是北极星" |
```

### 5.2 结尾金句模板

结尾金句要满足：
- **简洁**：15字以内效果最好
- **有对比**：A vs B 结构
- **可引用**：拿出来能独立传播
- **不说教**：是洞察，不是建议

**参考示例**：
- 「承诺可以堆砌，信任只能挣来。」（参考文档）
- 「让人停下来说'你一定要用这个'的那一刻，就是AI产品成功的信号。那个瞬间，不能买来，只能挣来。」（参考文档）
- 「先做深度，再做广度。让每一个广度，都有深度。」（参考文档）

## 六、彩蛋章节写法

> 彩蛋 = 林克人格化表达 + 自我印证

**彩蛋的作用**：
1. 降低文章的「说教感」
2. 让读者理解「写这篇文章的是谁」
3. 将全文论点映射回 CF，增强可信度（自我印证）
4. 增加传播性（人格化内容更容易被分享）

**彩蛋写法模板**：

```markdown
# 08 彩蛋

[AI生图]

写完[文章主题]，我意识到一件有意思的事——

**[连接点：写文章的工具/方法本身，就是全文论点的注脚。]**

**[CF/林克 是怎么做的？]**

它起步于一个极其聚焦的场景：[核心场景]。

不是[宽泛的描述]，不是[模糊的定义]——而是把[核心场景]这件事做到极致：
[具体描述做到极致的表现]……直到用了之后说出那句话："这个工具，我离不开了。"

做到这一步，才开始往外延展。

但有意思的地方来了：[核心场景]不只是一个功能，它是一种源（元）能力。

[元能力的层层衍生：A → B → C → D……]

某种意义上，它就是这套理论的自我实证：
先在一个地方挣到信任，再把这份信任延伸到更多地方。

📌 **[结尾金句]**

---

如果你想了解林克对AI领域的更多洞察，欢迎访问 [AI洞察首页](https://xiaoxiong20260206.github.io/ai-insight/) 了解更多。
```

**彩蛋写作注意**：
- 语气比正文轻松，可以用「有意思的事」「某种意义上」等口语
- 「连接点」要真实，不能强行拉扯（论点和 CF 要有真实关联）
- 彩蛋中的「自我印证」要具体，不是「我们也做得很好」，而是「我们的具体做法验证了这个原理」
- 结尾金句是整篇文章的最后一个标点，要可以独立传播

## 七、AI生图章节 Prompt 规范

彩蛋章节配图使用 AI 生图，要求：

**风格关键词**（选2-3个）：
- cyberpunk、futuristic、holographic（科技感）
- watercolor、illustration、pastel（温暖感）
- minimalist、clean lines、geometric（简洁感）

**内容关键词**（根据主题选择）：
- AI产品主题：neural network, data flow, human-AI collaboration
- Claude Code主题：code terminal, circuits, developer workspace
- 信任/产品主题：handshake, growth, trust building

**禁止**：
- 禁止使用文字主导的图片（文字在AI生图中容易出错）
- 禁止生成特定真实人物形象
- 禁止深色系（KIM Doc中深色图片视觉割裂感强）

**Prompt 模板**：
```
[主题描述 in English], [风格关键词], light color scheme with white background, 
professional illustration, high quality, no text, 16:9 ratio
```

**示例**：
```
AI product quality threshold concept, cyberpunk minimal illustration, 
light pastel colors with white background, glowing threshold line separating 
failure zone and success zone, professional illustration, high quality, no text
```
