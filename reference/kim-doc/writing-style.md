### 0.4 表格格式：空列头风格，但分隔线后不要空行（沈浪修改定稿版确认）

**大多数表格用 `| | | |` 空列头风格**（从标杆B学习）：

```markdown
|  |  |  |
| --- | --- | --- |
| 内容A | 内容B | 内容C |
```

⚠️ **关键细节：分隔线 `| --- |` 后面直接是实际内容行，不要加空行 `| | | |`！**

我原来犯的错误——分隔线后多了一行空行：
```markdown
|  |  |  |
| --- | --- | --- |
|  |  |  |        ← ❌ 这一行是多余的！导致KIM Doc渲染多一行空行
| 两种理解 | 代表策略 | 结果 |
```

沈浪修改定稿版的正确格式——分隔线后直接是内容：
```markdown
|  |  |  |
| --- | --- | --- |
| 两种理解 | 代表策略 | 结果 |    ← ✅ 直接跟内容，不要空行
```

**但当第一列是分类维度时，用有意义的列头**：

```markdown
| 维度 | 可模仿的部分 | 不可模仿的部分 |
| --- | --- | --- |
| 制度 | DRI→业界传播 | 信念体系 |
| 角色 | FDE→全行业复制 | 世界观 |
```

**原理**：KIM Doc中空列头渲染更清爽，但当第一列本身就是维度分类时，空列头反而让人不知道每行代表什么。

**两列表格的格式**（从→到转变表）：

```markdown
|  |  |
| --- | --- |
| 旧时代规则 | AI时代新规则 |
| 护城河=技术专利 | 护城河=组织形态×数据×技术 |
```

同样：分隔线后直接跟内容，不要加空行。

### 0.5 📌标注规范

- 每章最多1-2次📌
- 📌只用于**反常识判断**或**关键结论**
- 格式：`📌 核心结论内容`（不加粗，让📌本身做标识）
- 不用于常识性陈述

### 0.6 正文章节：调研五步法（核心写作逻辑）

> **核心原则**：每个正文章节不是随意铺陈信息，而是遵循"表象→规律→类比→回验→预测"的调研逻辑，层层推进，让读者跟着你的思考链条走。

**五步法详解**：

| 步骤 | 做什么 | 为什么 |
|------|--------|--------|
| ① 表象扫描 | 列出你调研到的所有现象、事实、数据，不急于解释 | 先把"看得见的东西"摆全，不遗漏 |
| ② 本质提炼 | 从表象中提炼底层规律/驱动力/结构（运用本质洞察力） | 透过现象看本质，找出"看不见的手" |
| ③ 类比表达 | 用生活化类比把规律翻译成人话 | 读者能秒懂，而不是"看起来对但站不住" |
| ④ 回验表象 | 把类比/规律带回表象，检验是否匹配 | 验证规律的解释力——能解释多少现象？不匹配的是例外还是规律有问题？ |
| ⑤ 预测未来 | 基于规律推演未来发展趋势 | 规律的价值在于预测，不只是解释 |

**章节内的展开顺序**：
```
章节标题
配图
① 现象：A、B、C三个表象（有数据/事实支撑）
② 规律：这些表象背后的共同驱动力是X（📌标注关键判断）
③ 类比：这就像[生活化类比]，[类比展开]
④ 回验：用X回看A/B/C——A因为X→符合，B因为X→符合，C因为X→部分偏离（原因：...）
⑤ 预测：基于X，未来6-12个月可能出现D/E
```

**📌标注时机**：只在②本质提炼和⑤预测处使用📌，每章1-2次。

**子章节同样遵循五步法**，但可以更紧凑——现象和规律可以在同一段中给出，类比和回验可以合并。

**五步法的灵活性**：
- 不是每个子章节都5步全走完。如果一个子章节是补充案例，①④即可
- 但每个一级章节（`#`）必须完整走完5步
- ③类比是可选的——如果规律本身足够直观，不需要类比硬套

### 0.7 行动清单章节：P0/P1/P2分级+具体操作（日报型专用）

标杆A做法：
- 用🔴🟡🟢标注优先级
- 每条行动有"为什么值得做"的解释
- 最简版操作模板（四个问题，10分钟）

**深度调研型一般不含行动清单**，聚焦于解读和洞察。

### 0.8 总结章节：规律+现象+趋势的系统化总结

**不是简单的"从→到"转变表，而是对全文调研逻辑的系统性收束。**

总结章节必须包含三个部分：

| 部分 | 内容 | 格式 |
|------|------|------|
| 规律 | 全文提炼出的核心规律/驱动力是什么 | 1-2句话，用📌标注 |
| 现象 | 这些规律解释了哪些现象，哪些现象是例外 | 简表：现象→规律解释力（✅匹配/⚠️部分偏离/❌例外） |
| 发展趋势 | 基于规律的未来6-12个月预测 | 3-5条预测，每条有"因为X规律→所以Y"的逻辑链 |

**结尾金句**：最后一句必须是从规律中提炼的、可独立传播的洞察。简洁（15字内）、有对比、可引用、不说教。

### 0.13 KIM Doc图片处理方式（关键踩坑经验）

**核心问题**：docs-shuttle的push API推的MD文本里，CDN URL图片（`cdnfile.corp.kuaishou.com/kc/files/a/design-ai/...`）不会被KIM Doc自动转换为内部图片链接。KIM Doc只支持内部`is-docsfile`签名URL格式的图片。

**正确做法（按优先级排列）**：

1. **最佳方案：在KIM Doc编辑器里手动插入图片**
   - 打开KIM Doc编辑器 → 选择图片位置 → 点击插入图片按钮 → 上传本地图片或粘贴URL
   - KIM Doc编辑器会自动把外部图片转换为内部`is-docsfile`签名URL
   - 这是唯一100%可靠的方式

2. **次选方案：先push文本，再手动插入图片**
   - 先用push API推送纯文本内容（不含图片）
   - 然后在KIM Doc编辑器里逐个插入图片到对应位置
   - 适合图片数量不多（<15张）的场景

3. **临时方案：保留已有内部图片URL**
   - 如果文档已有图片（通过编辑器手动插入的），先pull文档获取`is-docsfile`签名URL
   - 把这些签名URL保存下来，替换kimdoc文件里的CDN URL
   - 然后push更新
   - **注意**：签名URL有7天过期时间，过期后需要重新pull获取新签名

**封面大图特殊格式**（沈浪修改定稿版确认）：
- 封面大图用 `[[封面：描述](CDN链接)](CDN链接)` 格式（双重括号，既是图片也是可点击链接）
- 普通章节配图用 `![图片](CDN链接)` 格式
- 但push API推送后，CDN链接图片都不会自动转换 → 需要在KIM Doc编辑器里手动插入图片

**图片格式对比**：

| 格式 | push后效果 | 说明 |
|------|-----------|------|
| `![图片](is-docsfile签名URL)` | ✅ 正常显示 | KIM Doc内部格式，编辑器自动生成 |
| `![图片](CDN URL)` | ❌ 显示失败 | push API不自动转换 |
| `[[图片](URL)](URL)` | ❌ 显示失败 | KIM Doc不支持此格式 |

**长期解决方案**：需要在docs-shuttle中增加图片上传API，把CDN URL图片先上传到Docs的图片存储服务，获取`is-docsfile`格式URL后再push。

## 一、整体风格定位

| 维度 | 规范 |
|------|------|
| **人称** | 第一人称（我、我们），偶尔「我们看到」「我认为」 |
| **立场** | 鲜明，有判断，不做「一方面另一方面」的两面派 |
| **语气** | 自信但不傲慢，严谨但不学究，幽默但不油腻 |
| **类比** | 大量使用生活化类比，让抽象概念具体可感 |
| **节奏** | 短段落为主（2-4句），重要观点单独一段加强 |
| **彩蛋** | 比正文轻松，允许口语化，要有「有意思」的发现感 |

## 二、文章开头写法

**好的开头 = 00概览章直接制造认知冲突**

文章不设引子章节，直接从00概览开始。00概览章承担"制造认知冲突"的功能：一句话定义事情本身→一个核心问题→范式对比表→📌剧透金句。

```markdown
# 示例

# 00 全文概览：先说结论
![概览配图](CDN链接)
一件正在发生的事：AI产品的外显部分正在全面趋同。
它带来了一个问题：当技术、功能和体验都差不多时，还剩什么？
[范式对比表]
📌 剧透：答案是组织形态——公司怎么组织人，才是抄不走的。
```

**禁止**：
- ❌ 以「在当今AI快速发展的时代」开头
- ❌ 以「本文将会介绍XXX」开头
- ❌ 以背景介绍/行业定义开头
- ❌ 使用📖引子章节（已废弃）

## 三、比喻使用规范

> 参考文档核心比喻：「买电动工具 vs 招员工」

**好比喻的标准**：
1. 来自日常生活（工具、员工、学生、餐厅…）
2. 映射关系清晰（A的属性 = B的属性）
3. 能衍生出多个子映射（一个比喻能解释多个现象）

**常用比喻类别**：

| 场景 | 可用比喻 |
|------|---------|
| AI产品 vs SaaS | 员工 vs 工具，摩托车 vs 自行车 |
| 质量阈值 | 菜品难吃 vs 刚刚好吃 vs 惊艳好吃 |
| 先推广再打磨 | 厨师还没学会做菜就开门迎客 |
| 场景聚焦 | 先把一个菜做到极致，再开第二道 |
| 信任积累 | 银行存款，不能立即提现 |
| 技术栈 | 编码是底层OS，其他能力是App |

**类比展开模板**：
```
[比喻名称] = [简要描述]

[具体展开：比喻是怎么运作的]

这个比喻，解释了[主题]所有的[特征]：

[对比表格 — 对比A和B在各维度的映射关系]
```

## 四、表格写法规范

### 4.1 表头要精简有力

| 差 | 好 |
|----|-----|
| 「用户的核心需求类型」 | 「用户要什么」 |
| 「产品失败的典型表现形式」 | 「典型错误」 |
| 「阶段性工作的关键核心原则」 | 「关键原则」 |

### 4.2 数据单元格要具体

| 差 | 好 |
|----|-----|
| 「有价值」 | 「直接省脑力，不用再想了」 |
| 「不好」 | 「会主动劝退，告诉周围人'不行'」 |
| 「重要」 | 「Day 1就要付费，顶级员工不需要试用期」 |

### 4.3 结论标识符（📌）用法

```markdown
# 正确用法 — 每章最重要的洞察，用📌标出

📌 AI产品有一条"质量阈值线"，低于阈值任何推广都是在加速负口碑传播。

# 对比规则：
- 每章最多用1-2次📌
- 📌后跟纯文本洞察，不加粗
- 📌标注的内容必须是「反常识的」或「关键判断的」，不是常识性陈述
- 📌格式：`📌 纯文本`，不是`📌 **加粗**`
```

### 4.4 产品评分表标准

```markdown
| 产品 | Q 价值密度 | E 成本收支 | T 信任 | D 聚焦 | 结论 |
|------|-----------|-----------|--------|--------|------|
| Cursor | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★★ | ✅ 教科书 |
| Claude Code | ★★★★★ | ★★★★ | ★★★★ | ★★★★★ | ✅ 教科书 |
| 某产品 | ★★★ | ★★ | ★★ | ★★ | ⚠️ 通用陷阱 |
```

**评分维度说明**：
- Q（质量/价值密度）：用完是否有「省了自己脑力」的感知
- E（经济性）：是否 Day 1 有付费，单位成本 vs LTV 是否合理
- T（信任）：是否有个性化记忆、是否有口碑传播
- D（聚焦度）：核心场景是否足够深，不是广铺功能

**结论标识**：
- ✅ 教科书
- ✅ 方向正确
- ⚠️ 验证了需求（但还有问题）
- ⚠️ 通用陷阱
- ❌ 需要反思

## 五、总结章节写法

> 详见 §0.8。深度调研的总结 = 规律 + 现象 + 发展趋势，不是简单的"从→到"表。

### 5.1 规律收束

用1-2句话把全文提炼的核心规律/驱动力说出来，📌标注。

### 5.2 现象回验

简表：把全文涉及的关键现象逐一标注规律解释力（✅匹配/⚠️部分偏离/❌例外）。诚实标注例外是严谨性的体现。

### 5.3 发展趋势

基于规律的3-5条未来预测，每条必须有"因为X规律→所以Y趋势"的逻辑链。

### 5.4 结尾金句

结尾金句要满足：
- **简洁**：15字以内效果最好
- **有对比**：A vs B 结构
- **可引用**：拿出来能独立传播
- **不说教**：是洞察，不是建议

**参考示例**：
- 「承诺可以堆砌，信任只能挣来。」（参考文档）
- 「让人停下来说'你一定要用这个'的那一刻，就是AI产品成功的信号。那个瞬间，不能买来，只能挣来。」（参考文档）
- 「先做深度，再做广度。让每一个广度，都有深度。」（参考文档）

## 六、彩蛋章节写法

> 彩蛋 = 林克人格化表达 + 自我印证

**彩蛋的作用**：
1. 降低文章的「说教感」
2. 让读者理解「写这篇文章的是谁」
3. 将全文论点映射回 CF，增强可信度（自我印证）
4. 增加传播性（人格化内容更容易被分享）

**彩蛋写法模板**：

```markdown
# 彩蛋：写这篇调研的林克
![彩蛋配图](CDN链接)
写完[文章主题]，我意识到一件有意思的事——

[连接点：写文章的工具/方法本身，就是全文论点的注脚。]

[CF/林克 是怎么做的？]

它起步于一个极其聚焦的场景：[核心场景]。

不是[宽泛的描述]，不是[模糊的定义]——而是把[核心场景]这件事做到极致：
[具体描述做到极致的表现]……直到用了之后说出那句话："这个工具，我离不开了。"

做到这一步，才开始往外延展。

但有意思的地方来了：[核心场景]不只是一个功能，它是一种源（元）能力。

[元能力的层层衍生：A → B → C → D……]

某种意义上，它就是这套理论的自我实证：
先在一个地方挣到信任，再把这份信任延伸到更多地方。

📌 [结尾金句]
```

**彩蛋写作注意**：
- 语气比正文轻松，可以用「有意思的事」「某种意义上」等口语
- 「连接点」要真实，不能强行拉扯（论点和 CF 要有真实关联）
- 彩蛋中的「自我印证」要具体，不是「我们也做得很好」，而是「我们的具体做法验证了这个原理」
- 结尾金句是整篇文章的最后一个标点，要可以独立传播

## 七、AI生图章节 Prompt 规范

**所有章节和子章节配图均使用 AI 生图**，每张配图内容必须与对应章节内容保持一致。

**章节配图要求**：
- 一级章节（`#`）：高密度信息图（infographic），用 `designai-Infographic-image` 技能生成
- 二级子章节（`##`）：可以是信息图或概念插画，根据内容复杂度选择
- 彩蛋章节：概念插画风格，用 `designai-generate-image` 技能生成
- 封面大图：高密度信息图，一张图看清全文核心

**风格关键词**（选2-3个）：
- cyberpunk、futuristic、holographic（科技感）
- watercolor、illustration、pastel（温暖感）
- minimalist、clean lines、geometric（简洁感）

**内容关键词**（根据主题选择）：
- AI产品主题：neural network, data flow, human-AI collaboration
- Claude Code主题：code terminal, circuits, developer workspace
- 信任/产品主题：handshake, growth, trust building

**禁止**：
- 禁止使用文字主导的图片（文字在AI生图中容易出错）
- 禁止生成特定真实人物形象
- 禁止深色系（KIM Doc中深色图片视觉割裂感强）

**Prompt 模板**：
```
[主题描述 in English], [风格关键词], light color scheme with white background, 
professional illustration, high quality, no text, 16:9 ratio
```

**示例**：
```
AI product quality threshold concept, cyberpunk minimal illustration, 
light pastel colors with white background, glowing threshold line separating 
failure zone and success zone, professional illustration, high quality, no text
```
