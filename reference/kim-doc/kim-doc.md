# sl-ai-insight 子技能：KIM Doc 文章生成 (v3.0)

> **底层规范**：本文件已迁移通用写作规范到基础层 skill `sl-kim-doc-writer`。
> 执行 KIM Doc 写作时，**必须先加载** `sl-kim-doc-writer/SKILL.md`，获取完整的写作规范、自检清单和防乱码指南。
> 本文件只保留 sl-ai-insight 专属的工作流程、脚本调用和配置。

**加载基础规范（强制）**：
```
加载 sl-kim-doc-writer/SKILL.md
→ 完整写作规则：sl-kim-doc-writer/reference/writing-rules.md
→ 自检清单：sl-kim-doc-writer/reference/checklist.md
→ AI生图规范：sl-kim-doc-writer/reference/infographic-prompts.md
```

---

## 概述

将已完成的AI深度调研转化为符合快手内部 KIM Doc 规范的专栏文章。
严格对标参考文档的内容结构、视觉风格和写作风格。

**触发词**：「生成KIM Doc」「写成KIM文章」「输出KIM格式」「KIM文档」「发布到Docs」

---

## 工作流程

```
调研已完成
    ↓
Step 1: 提取关键信息（核心论点 + 核心数据 + 核心表格）
    ↓
Step 2: 按标准文档结构生成 Markdown
    ↓
Step 3: 为每个一级章节生成配图（designai-Infographic-image 技能）
    ↓
Step 4: 整合成完整 Markdown 文件
    ↓
Step 5: 上传到 KIM Docs（docs-shuttle 技能）
    ↓
Step 6: 输出文档链接
```

## Step 1: 提取关键信息

从已完成的调研中提取以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| **调研主题** | 调研的核心主题名称 | "Claude Code 技术深度解析" |
| **核心结论** | 用一句话或一个公式表达 | "工具链集成度 > 模型能力" |
| **主要争议/反差** | 引发读者思考的现象 | "为什么不投广告反而增长更快？" |
| **核心洞察列表** | 3-7个核心发现，每个可独立成章 | "工具调用 → 文件系统 → 网络请求" |
| **产品/案例对比** | 横向对比表（如适用） | Cursor vs Claude Code vs Copilot |
| **行动建议** | 读者看完后应该做什么 | "先把1个核心场景打透" |
| **彩蛋连接点** | 如何将结论连接回 CF/林克 | "而写这篇文章的工具本身..." |

## Step 2: 标准文档结构

> **强制参考**: 见 `references/article-structure.md`

### 文章标题格式

**两种场景**：

**场景A：专栏文章（林克人格化）**

```
【林克的AI洞察】[主标题：抓人眼球的核心观点]
```

- 前缀固定：`【林克的AI洞察】`
- 主标题：数字/比例 + 强烈判断/反差 + 悬念（效果最好）

**场景B：通用报告/分析文档**

```
[主标题：有立场有判断]——[副标题：补充范围]
```

- 主标题必须带立场：判断式、反常识式、类比式、冲突式（四选一）
- 副标题用 `——` 连接，补充范围/对象
- ❌ 禁止：`XX报告（YYYY年M月）`、`关于XX的分析与建议`、`XX使用体验分享`、`XX产品调研`

**命名规则**（两种场景通用）：
- 避免：过于学术的标题、没有立场的描述性标题
- 效果前置：前3秒定胜负，标题也要3秒定胜负
- 金句感：标题本身可独立传播

**判断方法**：写完内容后，提炼全文最重要的一个判断或洞察作为标题。

### 章节结构（标准模板）

**章节标题格式**：两位数字前缀 + 空格 + 章节名（两者缺一不可）

- ✅ `# 01 用户是谁，在干什么`、`# 02 痛点全景`、`# 04 对标业界`
- ❌ `# 一、背景`、`# 二、分析`（中文序号+泛化名）
- ❌ `# 引入`、`# 核心观点`（无序号+泛化名）

**章节层级**：最多两级——`# 02 章节名` + 粗体子段（如 `P0痛点——直接影响留存`），**不用###三级标题**。

**章节规划原则**（v2.0新增，最影响逻辑连贯性）：

1. 按读者理解路径排列，而非按信息类型分类
2. 同类信息合一章——如痛点不拆成"P0痛点""痛点分析""P1痛点"三章，合并为`# 02 痛点全景`
3. 每章只回答一个核心问题（"痛点有哪些？""好在哪里？""怎么改？"）
4. 章节数量6-8个，不超过10

```markdown
# [文章标题——副标题]

[一行平铺元信息：数据来源/日期/目标读者]

📌 核心判断：[用一行表达最重要的洞察]

[核心范式对比表格]

---

# 01 [引发思考的现象/反差]

[用1-2段描述让读者困惑的现象]
[数据/案例引入]

---

# 02-0N [核心观点/洞察章节]

[观点陈述]

[对比表格支撑论点]

📌 核心结论：[一句话点睛]

[延伸分析]

---

# [N-1] 总结

[三个转变表格：从→到]

📌 [金句]

---

# [N] 彩蛋

[AI生图：比喻插画]

[以第一视角，将文章结论映射回自身]

📌 [结尾金句]
```

## Step 3: 配图策略（v3.0 — 每章节必须有高密度信息大图）

> **⚠️ 公共规范**：配图的完整规范（质量标准/防乱码P0红线/何时配图）见 `reference/output-format-spec.md` §3.4。本文件只写AI洞察专属的配图策略。

**核心原则（v3.0变更）**：每个一级章节必须有1张高密度信息大图。这不是装饰——是30秒信息入口，确保读者一张图就能看清章节80%的信息量。

**旧版"可选配图/宁可0张"策略已废弃。新版：每个章节必须有图。**

**AI洞察专属配图模板**：

| 章节类型 | 推荐信息图结构 | 关键内容 |
|---------|----------------|---------|
| 全文概览 | 5板块/5趋势矩阵 | 核心结论 + 关键数据 + 板块划分 |
| 竞争格局 | 竞品对比矩阵/生态位图 | 公司名 + 核心优势 + 定位 |
| 产品分析 | 五维评分雷达/功能对比表 | 维度名 + 分值 + 关键差异 |
| 趋势研判 | 时间线+趋势箭头 | 时间节点 + 信号 + 方向 |
| 行业数据 | 数据瀑布/漏斗/柱状 | 数据点 + 变化方向 + 关键数字 |
| 深度论证 | 逻辑链/因果图/三层架构 | 证据 → 推理 → 结论 |
| 彩蛋比喻 | 比喻插画（designai-generate-image） | 生活类比 + 论点映射 |

### AI生图防乱码（P0红线）

**根因**：AI生图模型对复杂中文字符渲染不稳定。

**解决方案**（详见 output-format-spec.md §3.4）：
1. 中文标签极简（2-4字/条）
2. 结构用英文描述，中文只做标签
3. 金句≤15字
4. 生成后必须检查（read工具查看图片确认无乱码）
5. 乱码 → 减少中文 → 重生成 → 再检查
6. 最多重试3次，3次后 → 纯英文标签版本

## Step 4: 内容写作规范

> **完整写作规范见**: `references/writing-style.md`

### 4.1 核心写作原则

1. **先结论后论据**：每章第一句是结论，后面是支撑
2. **表格优先**：对比信息、列举信息优先用表格，禁用代码块展示内容
3. **📌标注核心点**：每章最重要的一个洞察用 `📌` 标注（📌后**不加粗**，📌本身就是视觉标记）
4. **林克有态度**：不是信息罗列，是「林克认为」「林克判断」
5. **类比增强理解**：用生活化类比（员工 vs 工具、摩托车 vs 自行车）
6. **彩蛋接地气**：彩蛋章节要把全文论点映射回 CF，自我印证
7. **段落不超过4行**：超过就拆，手机阅读优先
8. **每3-4段插入视觉元素**：表格/分割线/配图
9. **用户原话链接简化**：用 `[原文]` 而非 `[[原文](URL)]`

### 4.2 表格使用规范

使用频率：**极高**（参考文档8篇文章用了8张对比表格）

**常用表格类型**：

**类型1 - 核心范式对比表（全文概览必用）**：
```markdown
| 维度 | SaaS（旧范式） | AI产品（新范式） |
|------|----------------|-----------------|
| 用户要什么 | 能用这个功能 | 帮我把这件事做好 |
```

**类型2 - 错误模式清单**：
```markdown
| 错误 | 原因 | 典型表现 |
|------|------|---------|
| **错误1：堆功能** | SaaS逻辑迁移 | 发布会展示30个功能 |
```

**类型3 - 五维评估框架（产品评分）**：
```markdown
| 产品 | Q 价值密度 | E 成本收支 | T 信任 | D 聚焦 | 结论 |
|------|-----------|-----------|--------|--------|------|
| Cursor | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★★ | ✅ 教科书 |
```

**类型4 - 阶段路径表**：
```markdown
| 阶段 | 动作 | 关键原则 |
|------|------|---------|
| 阶段一 | 聚焦核心认知任务 | 只有一个，不是十个 |
```

**类型5 - 从→到转变表（结论章节）**：
```markdown
| 从 | 到 |
|----|-----|
| "我们的AI有XX功能" | "我们的AI在XX任务上比用户自己做得好" |
```

### 4.3 禁止事项（P0）

- ❌ **禁止用代码块展示内容结构**（KIM Doc 渲染为灰色等宽字体，视觉很差）
- ❌ **禁止用文字流水账列举步骤**（改用有序列表或表格）
- ❌ **禁止加文章目录/TOC**（KIM Doc 不需要目录）
- ❌ **禁止在文章开头放「编制/版本/数据来源」5行信息块**（元信息放末尾附录或一行平铺文本）
- ❌ **禁止全文没有一张表格**（表格是 KIM Doc 的核心可读性保障）
- ❌ **禁止配图冗余**（如果表格已覆盖内容，不要再加信息图重复同样的信息）
- ❌ **禁止📌标注加粗**（📌本身就是视觉标记，再加粗视觉过重）
- ❌ **禁止###三级标题**（最多两级：`# 02 章节名` + 粗体子段）

### 4.4 林克自我表达风格

参考 `references/writing-style.md`，核心要点：
- 用「我认为」「我判断」「我们看」增强主观立场
- 类比要具体、生活化（不用「象限」「矩阵」这类抽象词）
- 彩蛋段落口吻更轻松，允许「有意思的事」「某种意义上」等口语化表达
- 结尾金句要**可被引用**（简洁、有力、有画面感）

## Step 5: 上传到 KIM Docs

调用 `docs-shuttle` 技能完成上传：

```
use_skill(skill_name="docs-shuttle", reason="上传 AI 深度调研文章到 KIM Docs")
```

### 上传后手动样式设置（P0）

由于 Markdown 导入不支持样式，上传后需在 KIM Doc 编辑器中手动设置：

| 设置项 | 操作 |
|--------|------|
| 文档宽度 | 右上角「...」→ 页面设置 → 选择「宽版」 |
| 表头样式 | 选中表头行 → 居中 + 加粗 + 背景色（浅灰 #F5F5F5） |
| 固定首行 | 右键表格 → 固定首行 |
| 第一列强调 | 选中第一列 → 背景色（浅蓝 #E6F3FF）+ 加粗（可选） |

## 完成自检清单（P0，v2.0升级为15项）

每次推送 KIM Doc 之前，逐项检查：

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 标题 | 有判断/冲突/类比，非公文式？主标题不带序号？ |
| 2 | 章节标题 | 两位数字前缀+空格+内容名，无泛化名？ |
| 3 | 章节层级 | 最多两级，不用###三级标题？ |
| 4 | 章节结构 | 同类信息在同一章，逻辑连贯无断裂？ |
| 5 | 错字 | 全文grep排查？尤其音近字（痃→痛） |
| 6 | URL编码 | percent编码正确？（%E4%BD%93≠%E4%BD%53） |
| 7 | 表格 | 全文≥3张，表头精简，单元格具体？ |
| 8 | 📌标注 | 每章≤2次，📌后不加粗，只用于反常识洞察？ |
| 9 | 配图 | 每个一级章节有1张高密度信息大图？彩蛋有比喻插画？图做30秒入口而非冗余？ |
| 10 | 配图质量 | AI生图无乱码？（逐张read查看）中文标签≤4字？ |
| 11 | 用户原话 | 每条后有 `[原文]` 可跳转？ |
| 12 | 开头 | 元信息一行平铺文本，非引用块非5行公文？ |
| 13 | 结尾金句 | 15字以内、A vs B结构、可独立引用？ |
| 14 | P0禁止 | 无代码块展示结构、无TOC、无开头信息块、有表格、每个章节有信息图？ |
| 15 | 已推送 | 已上传到KIM Docs并提供链接？ |
