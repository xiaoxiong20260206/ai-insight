# sl-ai-insight 子技能：KIM Doc 文章生成 (v1.0)

> **标准参考文档**: 《【林克的AI洞察】可能99%的AI产品都做错了》
> **参考链接**: https://INTERNAL_DOCS/k/home/VbG1hV4Mo3F8/fcAANT5C1u-wk3Fe_nOVSiG4Y
> **核心原则**: 结构化 + 表格驱动 + 有观点 + 有温度（林克人格化）

---

## 概述

将已完成的AI深度调研转化为符合快手内部 KIM Doc 规范的专栏文章。
严格对标参考文档的内容结构、视觉风格和写作风格。

**触发词**：「生成KIM Doc」「写成KIM文章」「输出KIM格式」「KIM文档」「发布到Docs」

---

## 工作流程

```
调研已完成
    ↓
Step 1: 提取关键信息（核心论点 + 核心数据 + 核心表格）
    ↓
Step 2: 按标准文档结构生成 Markdown
    ↓
Step 3: 为每个一级章节生成配图（designai-Infographic-image 技能）
    ↓
Step 4: 整合成完整 Markdown 文件
    ↓
Step 5: 上传到 KIM Docs（docs-shuttle 技能）
    ↓
Step 6: 输出文档链接
```

---

## Step 1: 提取关键信息

从已完成的调研中提取以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| **调研主题** | 调研的核心主题名称 | "Claude Code 技术深度解析" |
| **核心结论** | 用一句话或一个公式表达 | "工具链集成度 > 模型能力" |
| **主要争议/反差** | 引发读者思考的现象 | "为什么不投广告反而增长更快？" |
| **核心洞察列表** | 3-7个核心发现，每个可独立成章 | "工具调用 → 文件系统 → 网络请求" |
| **产品/案例对比** | 横向对比表（如适用） | Cursor vs Claude Code vs Copilot |
| **行动建议** | 读者看完后应该做什么 | "先把1个核心场景打透" |
| **彩蛋连接点** | 如何将结论连接回 CF/林克 | "而写这篇文章的工具本身..." |

## Step 2: 标准文档结构

> **强制参考**: 见 `references/article-structure.md`

### 文章标题格式

```
【林克的AI洞察】[主标题：抓人眼球的核心观点]
```

**示例**：
- `【林克的AI洞察】可能99%的AI产品都做错了`（参考文档）
- `【林克的AI洞察】Claude Code为什么让工程师"离不开"了`
- `【林克的AI洞察】2026年最值得关注的AI产品路径`

**命名规则**：
- 前缀固定：`【林克的AI洞察】`
- 主标题：数字/比例 + 强烈判断/反差 + 悬念（效果最好）
- 避免：过于学术的标题、没有立场的描述性标题

### 章节结构（标准模板）

```markdown
# 00 全文概览

[核心结论一句话]

[核心对比表格：展示最重要的范式对比]

📌 **成功公式/核心判断**：[用一行表达最重要的洞察]

---

# 01 [引发思考的现象/反差]

[用1-2段描述让读者困惑的现象]
[数据/案例引入]
[为什么这个问题值得思考]

---

# 02-0N [核心观点/洞察章节]

[可选：章节题图]

[观点陈述]

[对比表格支撑论点]

📌 **核心结论**：[一句话点睛]

[延伸分析]

---

# [N-1] 总结

[三个维度/三个转变/三个建议]

[总结表格：从→到]

**[最重要的一句话，建议加粗]**

---

# [N] 彩蛋

[AI生成主题配图]

[以林克第一视角，将文章结论映射到 CF 自身]
[连接方式：「写这篇文章的工具本身，就是这套理论最好的注脚」]

📌 **结尾金句**

[了解更多模块]
```

## Step 3: 图片生成规范

> **强制规则**：每个一级章节（`# 0N xxx`）开头**必须**配图，无例外。

### 图片类型

| 章节类型 | 推荐图表类型 | 生成方式 |
|----------|-------------|---------|
| 00 全文概览 | 不配图（直接用标题图） | — |
| 01 现象/反差引入 | 数据对比卡片、冲击性数字 | designai-Infographic-image |
| 02-0N 核心洞察 | 概念图、架构图、进化阶段图 | designai-Infographic-image |
| 总结章节 | 三维对比图、框架图 | designai-Infographic-image |
| 彩蛋 | AI主题插图（赛博/机械/未来感） | designai-generate-image |

### 图片生成调用方式

```
# 非彩蛋章节：调用 designai-Infographic-image 技能
use_skill(skill_name="designai-Infographic-image", reason="生成章节配图")

# 彩蛋章节：调用 AI 生图技能（英文 prompt）
use_skill(skill_name="designai-generate-image", reason="生成彩蛋章节插图")
```

### 图片风格要求（P0强制）

- **必须使用浅色系**：白底/浅灰底 + 深色文字
- **禁止深色主题**（KIM Doc 中深色图片视觉割裂感强）
- **图片尺寸**：宽 1060px，高度按内容自适应
- **清晰度**：deviceScaleFactor = 2（2x 清晰）

## Step 4: 内容写作规范

> **完整写作规范见**: `references/writing-style.md`

### 4.1 核心写作原则

1. **先结论后论据**：每章第一句是结论，后面是支撑
2. **表格优先**：对比信息、列举信息优先用表格，禁用代码块展示内容
3. **📌标注核心点**：每章最重要的一个洞察用 `📌 **核心结论**：` 格式标注
4. **林克有态度**：不是信息罗列，是「林克认为」「林克判断」
5. **类比增强理解**：用生活化类比（员工 vs 工具、摩托车 vs 自行车）
6. **彩蛋接地气**：彩蛋章节要把全文论点映射回 CF，自我印证

### 4.2 表格使用规范

使用频率：**极高**（参考文档8篇文章用了8张对比表格）

**常用表格类型**：

**类型1 - 核心范式对比表（全文概览必用）**：
```markdown
| 维度 | SaaS（旧范式） | AI产品（新范式） |
|------|----------------|-----------------|
| 用户要什么 | 能用这个功能 | 帮我把这件事做好 |
```

**类型2 - 错误模式清单**：
```markdown
| 错误 | 原因 | 典型表现 |
|------|------|---------|
| **错误1：堆功能** | SaaS逻辑迁移 | 发布会展示30个功能 |
```

**类型3 - 五维评估框架（产品评分）**：
```markdown
| 产品 | Q 价值密度 | E 成本收支 | T 信任 | D 聚焦 | 结论 |
|------|-----------|-----------|--------|--------|------|
| Cursor | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★★ | ✅ 教科书 |
```

**类型4 - 阶段路径表**：
```markdown
| 阶段 | 动作 | 关键原则 |
|------|------|---------|
| 阶段一 | 聚焦核心认知任务 | 只有一个，不是十个 |
```

**类型5 - 从→到转变表（结论章节）**：
```markdown
| 从 | 到 |
|----|-----|
| "我们的AI有XX功能" | "我们的AI在XX任务上比用户自己做得好" |
```

### 4.3 禁止事项（P0）

- ❌ **禁止用代码块展示内容结构**（KIM Doc 渲染为灰色等宽字体，视觉很差）
- ❌ **禁止用文字流水账列举步骤**（改用有序列表或表格）
- ❌ **禁止加文章目录/TOC**（KIM Doc 不需要目录）
- ❌ **禁止在文章开头放「作者/版本/数据来源」信息块**
- ❌ **禁止全文没有一张表格**（表格是 KIM Doc 的核心可读性保障）

### 4.4 林克自我表达风格

参考 `references/writing-style.md`，核心要点：
- 用「我认为」「我判断」「我们看」增强主观立场
- 类比要具体、生活化（不用「象限」「矩阵」这类抽象词）
- 彩蛋段落口吻更轻松，允许「有意思的事」「某种意义上」等口语化表达
- 结尾金句要**可被引用**（简洁、有力、有画面感）

## Step 5: 上传到 KIM Docs

调用 `docs-shuttle` 技能完成上传：

```
use_skill(skill_name="docs-shuttle", reason="上传 AI 深度调研文章到 KIM Docs")
```

### 上传后手动样式设置（P0）

由于 Markdown 导入不支持样式，上传后需在 KIM Doc 编辑器中手动设置：

| 设置项 | 操作 |
|--------|------|
| 文档宽度 | 右上角「...」→ 页面设置 → 选择「宽版」 |
| 表头样式 | 选中表头行 → 居中 + 加粗 + 背景色（浅灰 #F5F5F5） |
| 固定首行 | 右键表格 → 固定首行 |
| 第一列强调 | 选中第一列 → 背景色（浅蓝 #E6F3FF）+ 加粗（可选） |
