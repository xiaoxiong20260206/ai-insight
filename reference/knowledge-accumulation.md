## 概述

知识沉淀是AI-Insight的核心能力之一，目标是将每日调研、深度研究和用户分享的文章系统化地积累成可复用的知识体系。

**核心原则: 多角度提取** — 同一知识源可以按不同维度同时沉淀到对应的知识结构里。

## 知识时效性管理（v2.1新增，P1强制）

> **教训**: 2026-03-10日报因搜索关键词配置过时导致使用旧闻。知识库同样面临时效性风险：沉淀的"最佳实践"可能已被新发展推翻，"竞争格局"可能已经变化。

### 知识的时效性分类

| 知识类型 | 时效性 | 更新频率 | 风险 |
|---------|--------|---------|------|
| **核心原理** | 长期有效 | 低 | 几乎无 |
| **最佳实践** | 中期有效 | 每季度检查 | 新工具/方法出现后可能过时 |
| **竞争格局** | 短期有效 | 每月检查 | 公司更名/合并/停运 |
| **产品功能** | 短期有效 | 发现即更新 | 功能迭代快 |
| **融资估值** | 时点数据 | 每次使用前验证 | 数据快速变化 |

### 知识沉淀时效性规则

**P0规则**:
1. **标注时间戳**: 每条知识必须标注沉淀日期和信息来源日期
2. **过时知识处理**: 发现知识过时时，要么更新，要么标注"⚠️ 可能过时，待验证"
3. **周期性检查**: 最佳实践类知识每季度检查一次是否仍然有效

**自检清单**:
- [ ] 沉淀的知识是否仍然有效（未被新发展推翻）？
- [ ] 是否标注了知识的时间戳/版本？
- [ ] 是否有提醒机制检查知识是否过时？
- [ ] 引用旧知识时，是否先验证其当前有效性？

**与日报/周报/深度研究的时效性对比**:

| 任务类型 | 时效性窗口 | 时效性要求 |
|---------|-----------|-----------|
| 日报 | 当天 | 头条必须是今天的新闻 |
| 周报 | 本周 | 内容必须来自本周日报 |
| 深度研究 | 最近1-3个月 | 竞争格局信源≤3个月 |
| 知识沉淀 | 长期有效（但需标注日期） | 引用前需验证当前有效性 |

## 知识来源与沉淀规则

### 1. AI日报（每日自动）

日报生成后，按维度分流提取：

| 板块 | 提取内容 | 沉淀位置 |
|------|---------|---------|
| **大模型** | 新模型发布、技术突破、评测数据、能力演进 | `01-models/` |
| **AI Coding** | Agent框架、编程工具更新、开发实践 | `02-agents/` |
| **AI 应用** | 产品动态、应用案例 | `concepts/applications/` |
| **AI 行业** | 公司融资、战略动态、竞争格局 | `03-ai-companies/` |
| **企业AI转型** | 转型案例、提效数据、组织变革 | `04-enterprise-ai/` |
| **趋势洞察** | 趋势判断、行动建议 | `insights/weekly/` |

**沉淀阈值**: 只沉淀标记为 🔴 的重要资讯和有深度洞察价值的内容

### 2. 深度调研（完成后自动，多角度沉淀）

> **核心**: 一次深度调研可能同时产出4-5篇不同维度的知识文档，这是预期行为。

| 调研类型 | 提取维度 | 沉淀位置 |
|---------|---------|---------|
| **人物追踪** | 思想体系、核心观点 | `entity-profiles/people/` |
| **公司调研** | 公司画像(战略、产品) | `03-ai-companies/` 或 `entity-profiles/companies/` |
| **公司调研** | 技术架构(模型、Agent) | 按技术分类到 `01-models/` 或 `02-agents/` |
| **趋势分析** | 模型演进趋势 | `01-models/` |
| **趋势分析** | Agent发展趋势 | `02-agents/` |
| **趋势分析** | 原理洞察 | `insights/` |
| **技术调研** | 技术概念 | `concepts/` |
| **技术调研** | 最佳实践 | `best-practices/` |

#### 多角度提取示例：字节AI开挂指南

同一套调研页面，按不同维度提取：

| 维度 | 提取角度 | 沉淀文件 |

### 3. 用户分享文章（手动触发）

用户通过以下方式触发：
- "学习这篇文章"
- "帮我把这个沉淀到知识库"
- "这个观点很好，记下来"

处理流程：
1. 识别文章主题和类型
2. 判断涉及哪些维度（可能多个）
3. 按维度分别提取核心观点
4. 追加到对应维度的知识文件

## 多角度提取工作流

```
同一知识源
    │
    ├─→ 模型角度 ────→ 01-models/
    │   (技术演进、能力对比、厂商分析)
    │
    ├─→ Agent角度 ───→ 02-agents/
    │   (架构设计、开发实践、工具方法)
    │
    ├─→ AI企业角度 ──→ 03-ai-companies/
    │   (公司布局、商业模式、竞争格局)
    │
    └─→ 企业AI角度 ──→ 04-enterprise-ai/
        (转型路径、落地实践、组织变革)
```

### 提取原则

1. **完整性**: 每个维度独立成文，不依赖其他维度文档
2. **聚焦性**: 每篇文档只关注该维度的知识，避免内容发散
3. **可追溯**: 每篇文档标注来源，方便回溯原始材料
4. **无损提取**: 宁可多提取，不漏重要知识点

## 更新原则

### 增量更新

新知识追加到现有文件，保留历史内容：
- 在已有章节末尾追加新内容
- 标注更新时间和来源
- 不要覆盖原有内容

### 版本记录

重大更新需更新版本号和时间：
- 小更新：v1.0 → v1.1
- 大更新：v1.x → v2.0

### 交叉引用

相关概念间建立链接：
```markdown
参见: [Agent架构与设计模式](../02-agents/agent-architecture.md)
```

### 来源标注

所有知识点必须标注来源：
- 日报来源：`来源: AI日报 2026-03-04`
- 调研来源：`来源: [调研标题](/02-deep-research/...)`
- 文章来源：`来源: [文章标题](链接)`

### 索引同步

每次新增文档后，必须更新对应维度的 `README.md` 索引文件。

### Ingest Ripple（v4.0新增 — Karpathy LLM Wiki 吸收）

> **核心原则**：Ripple 加的是路标，不是行李。路标过期了可以拆，行李只会越堆越重。

知识沉淀完成后，不是结束——还要级联更新所有关联节点：

**L1 索引层（每次必做，成本最低）**：
- `knowledge/packages/{domain}/INDEX.md`：更新对应页面的一行摘要和更新时间
- `MEMORY.md`：加一行指针 `→ knowledge/packages/{domain}/xxx.md`

**L2 方法层（强关联时）**：
- 相关 skill SKILL.md：加一句 `相关洞察 → knowledge/packages/{domain}/xxx.md#section`
- 判断标准：新知识直接修正/补充了该 skill 的方法 → 触达；仅主题相关但不修正 → 不触达

**L3 内容层（强关联时）**：
- 关联 knowledge 文档：追加一个段落增量
- 判断标准：新知识直接修正/补充了该文档的核心论点 → 触达；弱关联 → 不触达

**绝不做的事**：
- ❌ 在 MEMORY.md 或 skill 文档中塞入大段原文（会膨胀）
- ❌ 每次都跑 L2/L3（弱关联不该触达）
- ❌ 复制原文到多个文件（只加指针，读者跳转看原文）

**配套机制**：引用活性 Lint（每周日心跳执行）
- 30天未被查询触动的指针 → 🟡 待审
- 90天未被触动的指针 → 🔴 可清理
- 清理 = 删除指针行，不删除原文档

## 知识库批量扩充方法论 (v3.0新增)

> **背景**: 2026-03-13实战验证。单次会话从67篇扩充到94篇，新增27个文件。
> 这是一套可复用的知识库系统性扩充工作流。

### 第一步: 全面盘点 (Audit)

```bash
# 1. 列出所有现有文档
find 04-knowledge-base -name "*.md" | sort

# 2. 统计各维度文档数
find 04-knowledge-base/entity-profiles/companies -name "*.md" | wc -l
find 04-knowledge-base/entity-profiles/people -name "*.md" | wc -l
find 04-knowledge-base/concepts -name "*.md" | wc -l

# 3. 检查空目录（说明有缺口）
find 04-knowledge-base/concepts -type d -empty
```

**缺口识别清单**:
- [ ] 哪些重要AI公司没有画像？
- [ ] 哪些关键人物缺失？
- [ ] 哪些概念子领域目录是空的？
- [ ] README索引与实际文件是否一致？

### 第二步: 批量信息收集 (Research)

**关键技巧: 并行搜索**

一次发起8-10个并行Web搜索，高效收集信息：

```
# 搜索模式: "[公司/人物名] 2026 latest [关键词]"
# 中国公司用中文搜索: "[公司名] 2026 最新动态 [关键词]"
```

**搜索要素矩阵**:
| 目标类型 | 搜索关键词模板 |
|---------|--------------|
| 公司画像 | `[公司名] 2026 valuation funding revenue latest` |
| 人物画像 | `[人物名] 2026 latest [公司名]` |
| 中国公司 | `[公司名] 2026 最新动态 融资 估值 用户数` |
| 技术概念 | `[概念名] architecture landscape comparison 2026` |

### 第三步: 批量创建 (Create)

**关键技巧: 一次创建3个文件**

使用 `write_to_file` 并行创建多个文件：
- 每批次3个文件（工具并发限制）
- 每个文件使用标准化模板（见下方升级版模板）
- 先创建所有文件，再统一提交

### 第四步: 提交推送 (Commit)

**Git提交规范**:
```
知识库[扩充类型] Phase N: X个新文件

新增公司画像 (N家):
- [公司1]: [一句话描述]
- [公司2]: [一句话描述]

新增人物画像 (N位):
- [人物1]: [一句话描述]

画像总数: 人物X->Y、公司X->Y
```

### 第五步: 索引同步 (Index Sync)

**P0: 三层索引必须全部更新**

| 层级 | 文件 | 更新内容 |
|------|------|---------|
| L1: 总索引 | `04-knowledge-base/README.md` | 版本号、changelog、人物表、公司表 |
| L2: 维度索引 | `03-ai-companies/README.md` 等 | 画像列表、人物列表、待填充列表 |
| L3: 统计数据 | 各处人物/公司计数 | 所有出现数字的地方 |

**索引同步自检**:
- [ ] README.md 版本号已更新
- [ ] README.md changelog已添加本次记录
- [ ] 人物画像表已更新（数量+新增行）
- [ ] 公司画像表已更新（数量+新增行）
- [ ] 维度README(03-ai-companies等)已更新
- [ ] "待填充"列表已移除已完成项
- [ ] 所有链接路径正确（相对路径）
