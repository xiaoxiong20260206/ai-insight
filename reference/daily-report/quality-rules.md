## 核心原则

### 链接质量三角

```
         可达性
          /\
         /  \
        /    \
       /      \
      /________\
   新鲜度    权威性

三者必须同时满足，缺一不可。
```

## 时间窗口定义 (v4.0 P0)

### N日日报 = 收录 N-1日08:00 ~ N日08:00 的内容

```
例: 3月11日日报
├── 时间窗口: 2026-03-10 08:00 ~ 2026-03-11 08:00 (24小时)
├── 收录: 3月10日下午的产品更新、3月10日晚间的融资新闻、3月11日凌晨的公告
└── 不收录: 3月10日08:00前的内容（属于3月10日日报）
```

### 时间窗口 vs 新鲜度

| 概念 | 定义 | 用途 |
|------|------|------|
| **时间窗口** | N-1日08:00 ~ N日08:00 | 判断"这条新闻属于N日日报吗" |
| **新鲜度** | 日报日期±7天内 | 判断"这个链接是否过时" |

两者必须同时满足。

## 链接可达性 (v4.2 P0)

### 禁止项
- `#` 占位链接
- 空链接
- `javascript:void(0)`
- 非 http/https 开头的链接

### 验证方法

```
方式A (推荐): fetch_web 访问链接
方式B (快速): browser_agent 点击验证
方式C (抽样): 至少验证20%，重点验证头条和按钮链接
```

### 失效处理

| 情况 | 处理 |
|------|------|
| 404 | 搜索替代来源 或 archive.org |
| 域名错误 | 修正拼写 |
| 路径不完整 | 补全或用官网首页 |
| 需登录 | 标注或找公开替代 |

## 数据源平台标注 (v4.5 P0)

### 规则
当新闻条目来源于微信公众号或小红书数据源时，必须在source字段中添加平台标注。

### 格式

| 数据源 | source字段标注 | 链接处理 |
|--------|---------------|---------|
| 微信公众号 | 加 **(微信)** | 搜公开转载链接(36kr/huxiu等)，找不到用搜狗搜索URL，再找不到就不放链接（见封闭平台处理） |
| 小红书 | 加 **(小红书)** + 热度数据 | url填公开替代链接，xhs_url填 `explore/<noteId>` 永久链接（已验证永久有效） |
| Web/官媒 | 无需额外标注 | 直接使用原始URL |

### 示例
```
# 微信来源 (有公开替代链接)
"source": "NVIDIA/多源 · 3月13日(微信)",
"url": "https://blogs.nvidia.com/blog/nemotron-3-super-agentic-ai/"

# 微信来源 (找不到公开链接 → 不放链接)
"source": "某公众号 · 3月13日(微信)",
"url": ""

# 小红书来源(附带热度数据)
"source": "36氪 · 3月13日(小红书)| 3794赞6516收藏",
"url": "https://36kr.com/p/xxx"

# 纯Web来源(无需标注)
"source": "VentureBeat · 3月13日"
```

## 封闭平台链接处理 (v6.0 P0 — 2026-03-15 深度验证版)

### 第一原则: 错误的链接比没有链接更糟糕
> 用户点了个首页，不知道发生什么，丧失信任。
> 标对标题 + 标明来源(微信/小红书)，至少用户可以自己去搜。

  ├── weixin.sogou.com/link?url=... (跳转链接，数小时过期)

### 决策速查表

| 来源 | 推荐方案 | url字段填写 | 备注 |
|------|----------|------------|------|
| 微信(有转载) | ✅ 公开转载 | 36kr/huxiu链接 | 优先选择 |
| 微信(无转载) | ✅ 搜狗搜索 | weixin.sogou.com/weixin?type=2&query= | 永久稳定 |
| 微信(都不行) | ✅ 不放链接 | "" | 信息价值 > 链接 |
| 小红书 | ✅ noteId | explore/<noteId> | 永久有效(已验证) |

### 平台特性与链接能力 (2026-03-15 实测)

| 平台 | 永久链接可用性 | 说明 |
|--------|---------------------|------|
| **微信公众号** | ❌ 无法从外部获取永久URL | 微信2016年起从搜狗代理页清除`sn`参数，browser_agent获取的`src=11&timestamp`格式约6h过期 |
| **小红书** | ✅ `explore/<noteId>` 永久有效 | noteId是笔记唯一ID，无需登录即可阅读内容（登录弹窗不影响阅读） |

### 微信链接处理 (三级降级)

```
路径A (最佳): 找到公开转载链接 → 用作 url 字段
  Step 1: search_web("文章标题" site:36kr.com OR site:huxiu.com OR site:51cto.com)
  Step 2: fetch_web(候选URL) → 验证内容与标题叙事一致
  Step 3: url字段填入该链接，source字段标注(微信)

路径B (兜底): 用搜狗搜索URL
  → url字段填: https://weixin.sogou.com/weixin?type=2&query={url_encoded_title}
  → source字段标注(微信)
  → 此URL永久稳定，读者可从搜索结果找到原文

路径C (最终兜底): 不放链接，仅标注来源
  → url字段留空或设为""
  → 标题写准确完整，source标注(微信)+公众号名称
  → 信息价值 > 链接完整性
  → 旧版send_ai_daily.py预检已兼容空URL（脚本已废弃，质量门由daily_quality_gate.py独立执行）(v2026-03-15修复)
```

### 小红书链接处理 (2026-03-15 深度验证)

```
小红书笔记有永久链接！

链接格式: https://www.xiaohongshu.com/explore/<noteId>
来源: search.sh 返回的 feed["id"] 就是 noteId

输出规则:
  - url 字段: 填写同一事件的公开权威替代链接(如官方博客、媒体报道)
  - xhs_url 字段: 填写 explore/<noteId> 永久链接
  - source 字段: 标注(小红书) + 热度数据

验证注意:
  ✅ browser_agent 打开 → 能看到内容 = 有效
  ❌ fetch_web 访问 → 返回404模板页 (假阴性! Vue.js SPA)

已验证永久链接示例:
  explore/69965d4c000000000b013b86 → 正常渲染
  explore/6978d2e4000000002200b559 → 正常渲染
  explore/69a2c330000000001a0300c9 → Vibe Coding 1616赞
```

> **以终为始（虓晗教导 2026-03-15）**: 不能因为链接拿不到就换凑数信息。信息价值 > 链接完整性。

### 绝对禁止: 放一个"错"的链接
```
❌ 用通用首页凑数 (venturebeat.com/, github.com/)
❌ 用列表/分类页凑数 (developer.nvidia.com/blog/)
❌ 用内容不匹配的文章凑数 (标题说Vibe Coding，链接是Vibe Searching)
❌ 微信临时URL (mp.weixin.qq.com/s?src=11&timestamp=...) — 约6h过期
❌ 搜狗跳转链接 (weixin.sogou.com/link?url=...) — 数小时过期
❌ 编造看似真实的出版物URL (domain真+path假, 如nytimes.com/2026/.../fake-slug.html)
   → 高级伪造：通过格式检查，只有fetch_web才能发现404 (经验36 v2)
```

## 大事件分流搜索协议 (v9.0 P0 — 2026-03-17复盘新增)

> **教训 (2026-03-17)**: GTC 2026主导当天所有搜索，导致：
> - AI Coding板块0条新闻（完全空白）
> - NVIDIA单一源占比55%（6/11条）
> - 小红书搜索完全跳过
> - 板块间内容量极度失衡
>
> **根因**: 大事件形成"隧道视野"——所有注意力和搜索都聚焦于一个事件/公司，
> 其余板块在"主事件搜完已占满context"后被压缩或跳过。

### 触发条件
当Step 0.5探针识别到**某单一事件/公司/话题占据热点3条以上**时，自动启用本协议。

### 分流搜索规则 (P0)

```
大事件识别 (Step 0.5热点探针 ≥3条指向同一事件)
  │
  ├─ Phase 1: 大事件搜索 (≤4条精选)
  │     └─ 只允许该事件占据最多4条新闻
  │
  ├─ Phase 2: 独立板块搜索 (P0 不可跳过!)
  │     ├─ 对每个非主事件主导的板块，独立执行搜索
  │     ├─ 搜索关键词必须排除大事件关键词
  │     ├─ 例: GTC日 → AI Coding搜索 "Cursor OR Copilot OR Claude Code -GTC -NVIDIA"
  │     └─ 每个板块至少产出1条独立新闻
  │
  ├─ Phase 3: 小红书/微信独立搜索 (P0 不可跳过!)
  │     ├─ 微信: 双轨搜索(账号+话题)照常执行
  │     └─ 小红书: 至少搜索3个关键词
  │
  └─ Phase 4: 板块均衡自检
        ├─ 5个板块每个至少1条 (硬门控)
        ├─ 5个板块每个建议2条以上
        └─ 单一公司/来源 ≤ 40%
```

### 大事件vs日常对比

| 维度 | 日常模式 | 大事件模式 |
|------|---------|-----------|
| 搜索策略 | 5板块并行搜索 | Phase 1聚焦+Phase 2分流 |
| 单事件上限 | 无硬限制 | **≤4条** (含跨板块) |
| 板块覆盖 | 每板块≥2条 | 每板块**≥1条** (降级但不为0) |
| 源多样性 | 单域名≤40% | **同规则** |
| 小红书/微信 | 常规搜索 | **强制独立搜索** (不因大事件跳过) |

### 类比
> 暴风雨天气不代表只报暴风雨——天气预报仍然要告诉你明天气温、空气质量、穿衣建议。
> 大事件主导但不能垄断——读者需要的是全景视角，不是单一事件的N种角度。

## 五大板块分类

### 边界定义

| 板块 | 边界 | 典型内容 |
|------|------|---------|
| 🧠 大模型 | 基础模型能力、训练技术、模型发布 | GPT-5.4能力、DeepSeek V4发布 |
| ⌨️ AI Coding | 编程工具、IDE、代码助手 | Cursor更新、Claude Code ARR |
| 📱 AI应用 | 消费级产品、Agent应用、用户端产品 | ChatGPT周活、OpenClaw |
| 🏭 AI行业 | 投融资、估值、政策、市场格局 | OpenAI估值、VC投资总额 |
| 🔄 企业转型 | 企业AI落地、提效案例、组织变革 | 企业AI转型案例 |

### 分类原则
1. **按产品本质分类，不按信息来源分类**
2. **编程工具 vs 模型能力**: Claude Code→AI Coding, Claude Opus→大模型
3. **Agent产品 vs 模型**: OpenClaw→AI应用, Agent能力讨论→大模型
4. **用户数据归属**: ChatGPT用户数→AI应用（产品维度）

### 跨域去重
- 同一事件只出现在**最相关**的板块
- Cursor融资 → AI Coding（不是AI行业）
- OpenAI估值 → AI行业（不是大模型）

## 交叉验证 (v3.2)

### 必须验证的事实类型

| 类型 | 要求 |
|------|------|
| 产品发布 | 官网或官方社交媒体一手确认 |
| 融资金额/估值 | 至少2个独立来源 |
| 研究数据 | 回溯到原始报告/论文 |
| 排名/benchmark | 原始评测机构或论文 |
| 竞争格局 | 至少2个2026年内独立来源 |

### 验证方法
1. 一手源优先: 官网 > 论文 > 权威媒体 > 二手转述
2. 数字回溯: 每个关键数字追溯到原始来源
3. 时效性检查: 估值数据用最新报道时间

## ⛔ 修复策略三原则 (v8.3新增 — 经验37)

> **2026-03-16技能修炼发现**: 质量门报"9条新闻超出时间窗口"后，执行者选择
> "删除超窗内容"而非"搜索窗口内新闻替换"，导致内容从15条降到7条(-53%)。

### 三种修复策略对比

| 策略 | 做法 | 结果 | 判定 |
|------|------|------|------|
| **删减型** | 删除不合格内容 | 内容缩水，破坏完整性 | ❌ 禁止 |
| **绕过型** | 修改数据使其"通过" | P0违规，数据伪造 | ⛔ 红线 |
| **替换型** | 重新搜索合格内容替换 | 正确修复，保持内容量 | ✅ 正确 |

### 类比
> 病人发烧39°C，给他敷冰块降到35°C——症状消失了，但病人快冻死了。
> 正确做法是找病因对症下药，不是强行降温。
>
> **删除超窗新闻 = 敷冰块。搜索窗口内新闻替换 = 对症下药。**

### 正确的修复流程

```
质量门失败
  │
  ├─ 判断失败类型
  │     ├─ 时效性失败 → 回到Step 1搜索窗口内新闻**替换**
  │     ├─ 链接失败 → 搜索替代链接
  │     └─ 分类失败 → 调整分类或补充新闻
  │
  ├─ 修复后检查副作用
  │     ├─ 对比修复前后行数 (diff > 30% → 暂停)
  │     ├─ 对比修复前后新闻条数 (< 昨天70% → 报警)
  │     └─ 检查必需模块是否完整
  │
  └─ 内容量低于参考值70% → 报警，不继续
```

## ⛔ 经验40：WARNING→ERROR升级教训 (v9.1)

### 根因
2026-03-18复盘：分类错误、微信链接全空、小红书跳过——同一类问题反复出现。

**根本原因：WARNING级别的检查等于不存在。**

| 检查项 | 原级别 | 问题 |
|--------|--------|------|
| check_board_classification | WARNING | 分类错误也放行 |
| check_region_classification | WARNING | 地区错误也放行 |
| check_source_diversity | WARNING | 微信=0也放行 |
| 小红书覆盖 | 不检查 | 完全没有门控 |

### 死循环
```
问题发生 → 用户指出 → 沉淀记忆(文字) → 不改脚本级别
  → 下次执行 → 脚本仍放行 → 问题再次发生 → 用户再次指出...
```

### 解决方案 (v9.1)
| 检查项 | 原级别 | 新级别 |
|--------|--------|--------|
| check_board_classification | WARNING | **ERROR** |
| check_region_classification | WARNING | **ERROR** |
| check_source_diversity (微信) | WARNING | **ERROR** |
| check_source_diversity (小红书) | 不检查 | **v11.0: 不纳入日常，仅用户要求时搜索** (warning → skip) |
| check_source_diversity (集中度) | WARNING | **ERROR** |

### 核心原则
> **知道规则≠执行规则。WARNING≠ERROR。只有ERROR才能阻断部署。**
> **如果一个问题被用户反复指出，说明现有的检查级别（WARNING）是错误的。**
> **记忆是弱约束，脚本是强约束——但脚本的WARNING也是弱约束，只有ERROR才是真正的强约束。**
